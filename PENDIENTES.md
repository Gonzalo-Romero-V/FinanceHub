## Fase M/N/S — Mobile (Capacitor) + Notificaciones + Seguridad backend

Reemplaza por completo a las Fases E y F originales (alcance definido con el usuario, más amplio de lo previsto inicialmente). Orden explícito pedido por el usuario: **(1) completitud funcional mobile/web y su coexistencia → (2) notificaciones (deja lista la infraestructura de compilación Android) → (3) seguridad backend**. Las 3 partes están interconectadas (notificaciones nativas necesitan Capacitor andando; seguridad backend necesita saber qué orígenes nuevos aparecen) así que se documentan juntas pero se ejecutan en ese orden.

**Decisiones ya confirmadas con el usuario:**
- La app Android **no es un simple WebView remoto** — el diferencial de tener app es: (a) respaldo offline de movimientos (crear, NO editar/eliminar offline) con sincronización automática al reconectar, (b) notificaciones/recordatorios nativos. Esto implica **Next.js exportado como sitio estático** embebido en el APK (no SSR/rewrites locales), con capa de datos local (SQLite vía Capacitor) + cola de sincronización.
- Alcance de caché offline (solo lectura, última foto sincronizada): Cuentas + Balance General, Movimientos recientes (~30-60 días), Conceptos y Cuentas (necesario para el form), Presupuestos y Deudas.
- Micrófono: el código web actual (`voice-recorder-button.tsx`, `getUserMedia`/`MediaRecorder`) funciona sin cambios dentro del WebView de Android — no hace falta desacoplarlo, solo config nativa (permiso `RECORD_AUDIO` en el manifest).
- Notificaciones: sí requieren desacople real (Web Push API+Service Worker vs FCM vía Capacitor) — arquitectura: tabla `notifications` en backend como fuente de verdad + inbox in-app (bell icon, código 100% compartido) + push como "empujón" best-effort sobre esa fuente de verdad. Recordatorios de fecha conocida (reconciliación, cuotas de deuda) se agendan además como **notificaciones locales** (`@capacitor/local-notifications`, agendadas por el SO del dispositivo) para que disparen con cero conectividad ese día. Alertas reactivas (umbral de presupuesto) se evalúan contra caché local + cola pendiente cuando está offline, con el push del servidor como respaldo al reconectar.
- Disparadores de notificación confirmados: reconciliación próxima/vencida, alertas de presupuesto por umbral, cuotas de deuda próximas a vencer, límite de consumo de IA alcanzado, y aviso de sincronización completa al reconectar.
- Firebase/FCM: el usuario no tiene proyecto todavía — se crea cuando se llegue a esa parte puntual, con guía paso a paso (mismo patrón que Resend).
- Límite de uso de IA: ~50 consultas/día por usuario como punto de partida (ajustable).
